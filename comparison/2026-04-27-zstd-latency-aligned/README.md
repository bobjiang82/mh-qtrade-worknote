# 2026-04-27 zstd 第三轮 — 版本对齐 + 字节对齐 + 延迟分布

| 字段 | 9965 | 6979P |

| CPU | AMD EPYC 9965 | Intel Xeon 6979P |
| Host | 10.239.23.105 | 10.239.23.11 |

## 输入与 Reference frames（字节对齐）

- 输入语料：`silesia_shuf.tar`，**211,957,760 B (≈ 202 MiB)**
  - sha256：`ebfcba3c...d334400`（双机一致）
  - 注：上一轮 6979P 上的副本 sha256 不一致 (`79cd99...`)，本轮已重新 scp 对齐。
- Reference frames（在 9965 用 1.5.7 单线程 `-T1` 压出来，scp 到 6979P，sha256 双向校验一致）：

| frame | 字节 | sha256 (前 8) | Window Size | Block Max |
|---|---:|---|---|---|
| silesia.l1.zst  | 73,271,045 | `c575d10f` |   512 KiB | 128 KiB |
| silesia.l9.zst  | 59,192,801 | `52325f5e` |  4.0 MiB | 128 KiB |
| silesia.l19.zst | 52,861,396 | `c1c3f2de` |  8.0 MiB | 128 KiB |
| silesia.l22.zst | 52,270,427 | `6a2ae211` |  128 MiB | 128 KiB |

> Block max 是 zstd frame 内部每个 block 的上限，固定 128 KiB（`ZSTD_BLOCKSIZE_MAX`，所有 level/window 都一样）。Window Size 才随 level 变。`zstd -lv` 输出已 dump 到 raw-9965/info.txt。

## 测试方法

- 外层 wrap：`python -c "time.perf_counter"` 计时，每次跑前 `cat input > /dev/null` 预热 page cache，结果写到 `/dev/null`。
- 矩阵：
  - 压缩：L1 / L9 / L19 / L22 @ T=1，外加 L19 @ T=8
  - 解压：L1 / L9 / L19 / L22 @ T=1（解压本来就单线程）
- N=20 次 / 配置 / 机器，两机各 180 个数据点。
- 解压输入是上面字节相同的 reference frame；**两机解压的是完全同一字节流**。

## 关键结论

1. **解压两机几乎完全一致**（差距 < 1%，仅 L22 6979P 快 8%）。
   - 上一轮"9965 解压快 8%"是个伪结论，根因是两机用各自版本压出不同 frame（zstd 1.5.5 vs 1.5.1 切分微差）。本轮用同一字节流的 reference frame，差距消失。
2. **压缩 L19/L22：6979P 单核仍快 2%~8%**（L22 p50 80.3 s vs 87.1 s）。比上一轮报的"快约 20%"小很多——上一轮也是版本不同的污染。
3. **压缩 L1：6979P 快 31%**（0.45 vs 0.65 s）。低 level 短匹配 + entropy 主导，Xeon IPC/频率优势明显。
4. **压缩 L9：两机基本持平**（2.59 vs 2.69 s）。
5. **多线程压缩 L19 T=8：9965 反超 14%**（12.9 vs 14.7 s），且 std 远低（0.16 vs 1.37）。EPYC 9965 多线程一致性强，6979P 分布肥尾（p99 17.1 s）。
6. **延迟分布特点**：
   - 解压 std 都极小（< 0.01 s），稳定。
   - 压缩 6979P 在 L9/L19/L22 std 都比 9965 大 2~3 倍（系统噪声 / 同机其他负载）。
   - L22 T=1 最差情况 p99：6979P 91.9 s vs 9965 89.1 s，long tail 反而 6979P 更长。

## Caveats

- N=20，统计稳定性有限（特别是 p99）。L22 单次 80~90 s，提到 N=50 单这一档要 1 小时，时间换稳定性的取舍。
- 6979P 测试期间仍有约 408 GiB 内存被其它任务占用，部分压缩 std 偏大可能与背景负载相关。
- 多线程压缩 L19 T=8 9965 反超，要小心：silesia 211 MiB 在 T=8 已是饱和点附近，差距可能跟 worker 调度/缓存几何相关，不一定能外推到更大数据。
- 本轮没测大数据多线程 scaling 上限。
